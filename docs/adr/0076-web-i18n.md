# ADR-0076: Web internationalization — Russian default + Tajik (next-intl, cookie locale)

**Status:** Accepted
**Date:** 2026-06-04
**Implements:** UI localization request (post-Horizon) — make the interface
Russian, multilingual with Tajik.
**Builds on:** the Next.js web app + BFF (P0–P5), branding (P0.11 / ADR-0018).

## Context

The КЧС operator UI shipped **English-only** (all ~76 `.tsx` files hardcoded
English), while `<html lang>` was driven by `branding.localeDefault` (also
`"en"`). For a sovereign Tajikistan deployment this is wrong: the interface must
be **Russian by default** and, ideally, **multilingual with Tajik**. No i18n
framework was installed.

Forks locked with the user:
- **Approach:** `next-intl`, **no-routing setup** — the active locale lives in a
  cookie (`NEXT_LOCALE`), **not** a `/ru` URL prefix. Cleanest for an internal
  tool: existing routes, `typedRoutes`, and the auth middleware stay untouched;
  the locale is a user preference, not part of the URL space.
- **Languages:** **RU (default) + TG**. `localeDefault` semantics move to the
  cookie/`defaultLocale` (RU); Tajik is selectable via a topbar switcher.

## Decision

### Foundation (next-intl, cookie-based)

- `next.config.ts` wrapped with `createNextIntlPlugin("./src/i18n/request.ts")`.
- `src/i18n/config.ts` — `locales = ["ru","tg"]`, `defaultLocale = "ru"`,
  `LOCALE_COOKIE = "NEXT_LOCALE"`, an `isLocale()` guard.
- `src/i18n/request.ts` — `getRequestConfig` resolves the locale from the
  `NEXT_LOCALE` cookie (falling back to RU) and loads
  `messages/{locale}.json`. Works for **anonymous** requests too, so the login
  page is Russian out of the box.
- `src/i18n/locale-actions.ts` — a `setLocale(locale)` **server action** that
  writes the cookie (1-year, `sameSite=lax`).
- `messages/ru.json` + `messages/tg.json` — namespaced catalogs
  (`nav`, `topbar`, `login`, `locale`, `common`).
- Root `layout.tsx` — `<html lang={await getLocale()}>` and the body wrapped in
  `<NextIntlClientProvider locale messages>` (messages from `getMessages()`), so
  both server (`getTranslations`) and client (`useTranslations`) components
  resolve strings.
- `components/cmc/language-switcher.tsx` — a client `<select>` (RU/TG, shown in
  their native names) that calls `setLocale` then `router.refresh()`; mounted in
  the topbar.

### First slice (most-visible chrome + entry)

- **Sidebar** — all nav group + item labels keyed by their stable `id`
  (`getTranslations("nav")`), plus "Coming soon" / "Signed-out".
- **Topbar** — breadcrumb root, search placeholder, "Realtime", sign-out;
  switcher added.
- **Login** — mural ("ALL SYSTEMS OPERATIONAL", kicker/headline are still
  branding-driven), "Secure sign-in", "Welcome back", the AUP paragraph, and the
  form (email/password labels, "Need help?", "Continue"/"Signing in…", invalid
  credentials) — `metadata` title via `generateMetadata`.

Remaining pages are translated **incrementally** (tracked) — the infrastructure
makes each subsequent screen a "add keys + swap strings" change.

## Consequences

- **Positive:** Russian UI by default with one cookie flip to Tajik; no URL
  churn, middleware/routes/`typedRoutes` untouched; anonymous pages localized;
  ICU/plurals/interpolation available for future strings; server + client share
  one catalog.
- **Negative / trade-offs:** rollout is **incremental** — pages not yet migrated
  stay English until translated (tracked tasks #219–#221); catalogs are
  hand-maintained (no TMS yet); **branding copy** (orgName/headline/etc.) remains
  tenant-data-driven and separate from i18n (a tenant can still carry its own
  language there); locale is per-browser (cookie), not yet persisted to the user
  profile server-side.

## Validation

- Web **tsc ✓ + lint ✓** (no warnings/errors); dev server compiles clean with
  the plugin.
- **Live curl:** `/login` with no cookie → fully Russian, `<html lang="ru">`,
  zero leftover English (`Welcome back`/`Secure sign-in`/`Continue`/
  `ALL SYSTEMS OPERATIONAL` all gone); with `NEXT_LOCALE=tg` → Tajik, `<html
  lang="tg">`. Confirms cookie-based resolution and the switcher mechanism.
- Backend untouched → its suite unaffected.

## Status update (2026-06-04)

**Rollout complete.** All operator-facing pages + components are localized
(RU/TG) across slices 1–5: shell + login → dashboard + incidents + offline →
search + documents + map → admin/* → chat/wiki/video/media/monitoring/imports/
workflows/notifications + `pwa-register`. Catalogs hold **721 keys with exact
ru↔tg parity**; page breadcrumbs are localized. `room-stage` (LiveKit) and
`media-player` (HLS) carry no own UI strings. Each slice gated on tsc + lint +
`next build` + authed live-curl (RU default + `NEXT_LOCALE=tg`).
