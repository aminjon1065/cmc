# ADR-0019: RBAC ‚Äî per-tenant roles + global permission catalog

**Status:** Accepted
**Date:** 2026-05-25
**Implements:** PRIORITY_EXECUTION_PLAN P1.1 ‚Äî **first P1 item; the gate before any new domain module**
**Closes tech-debt:** TD-003
**Depends on:** ADR-0003 (RLS / tenant isolation), ADR-0008 (Redis), ADR-0011 (cache fail-open pattern)
**Unblocks:** P1.4 (Admin Panel), P1.5 (Incidents), every subsequent module

## Context

Until now tenant was the *only* access boundary: every authenticated user
in a tenant could read every document, and (once added) act on every
incident / case / workflow. TD-003 (S0) flagged this. The sequencing rule
is explicit ‚Äî **no new domain module before RBAC** ‚Äî because each module
added without it compounds the access-control debt. P1.1 lands the
backend: tables, a permission catalog, the `@Authorize` guard with a
Redis-cached permission set, seeded system roles, and the first protected
module (documents).

## Decision

### 1. Per-tenant roles, global permission catalog

- **`permissions`** is a GLOBAL catalog (no `tenant_id`) ‚Äî the platform's
  fixed `(domain, action)` capability list, the same for every tenant
  like an enum. RLS: readable by all (`FOR SELECT USING (true)`), writable
  only under `app.bypass_rls` (migrations/seed).
- **`roles`** are PER-TENANT (`tenant_id`, under RLS): each tenant owns
  its roles. System roles (`tenant_admin` / `operator` / `auditor`) are
  seeded per tenant and flagged `is_system` (immutable).
- **`role_permissions`** maps a role to catalog permissions; RLS scopes it
  via an `EXISTS` on the parent role's tenant (it has no `tenant_id` of
  its own).
- **`user_roles`** assigns a tenant role to a tenant user; `tenant_id`
  denormalised for a direct RLS predicate.

Per-tenant (not global) roles chosen so a tenant can grow its own roles
without touching others and the RLS pattern stays uniform with every
other table. The catalog is global because the *code* checks fixed
permission strings ‚Äî those can't differ per tenant.

### 2. The permission catalog + system roles live in `@cmc/contracts`

`PERMISSION_CATALOG` and `SYSTEM_ROLES` are the single source of truth for
both the seed (which writes them to the DB) and the guard (whose
`@Authorize('document:read')` strings reference them). They can't drift:
the seeded rows and the guarded strings come from the same constant. The
first catalog covers `document:{read,write,delete}`, `session:read`,
`audit:read`, `role:{read,assign}`.

System roles:
- `tenant_admin` ‚Üí every permission (`"*"`, resolved at seed)
- `operator` ‚Üí `document:{read,write}`, `session:read`
- `auditor` ‚Üí `document:read`, `session:read`, `audit:read`, `role:read`

### 3. `@Authorize(...)` guard, ALL-required semantics

`@Authorize('document:write')` on a handler (or `@Authorize(a, b)` for
all-of) sets reflector metadata; `AuthorizeGuard` enforces it:
- no metadata ‚Üí not permission-gated (pass; auth handled by JwtAuthGuard);
- metadata but no tenant context ‚Üí 401;
- otherwise ‚Üí `RbacService.enforce` throws **403** + a durable
  `rbac.access.denied` audit row on the first missing permission.

### 4. The guard-vs-interceptor ordering bug (load-bearing)

NestJS runs **guards before interceptors**. The `TenantTransactionInterceptor`
(which opens the request's tenant-scoped `app.tenant_id` transaction) has
*not run yet* when `AuthorizeGuard` fires. So the permission read **cannot**
use the ambient `tenantDb.run()` scope ‚Äî it doesn't exist at guard time
(this surfaced as a 500 "run() called outside a tenant scope").

`RbacService.resolvePermissions` therefore opens its **own**
`runForTenant(tenantId, ...)` transaction (it has the tenantId from the
JWT-derived context). RLS still confines the read. The admin mutation
paths (`assignRole` / `removeRole`), which run in *handler* context (after
the interceptor), keep using `.run()`. This split is documented in the
service.

### 5. Redis-cached permission set, fail-open + invalidated

`PermissionCacheService` caches the resolved `Set<Permission>` per
`(tenantId, userId)` at `cmc:authz:perms:<tenant>:<user>`, TTL
`RBAC_PERM_CACHE_TTL_SEC` (default 300s). Same posture as the session
cache (ADR-0011): **fail-open to DB** on any Redis error ‚Äî the DB is the
source of truth; the cache only optimises the per-request resolve.

Invalidation: `assignRole` / `removeRole` call `del(tenant, user)` after
the change commits; a role-level permission change would call
`delTenant(tenant)` (SCAN-based). Best-effort, bounded by the TTL. The
e2e suite proves an assign grants access *immediately* (cache invalidated)
and a remove revokes it.

### 6. Documents is the first protected module

`DocumentsController` now uses `@UseGuards(JwtAuthGuard, AuthorizeGuard)`
with `@Authorize('document:read')` on reads, `'document:write'` on
upload-init/finalize, `'document:delete'` on delete. This closes TD-003 in
practice, not just in infrastructure: a user with no role gets 403 on
every document route; an operator can read/write but not delete; an
auditor can only read.

### 7. Seeding + test fixtures share one path

`rbac-seed.ts` (`ensureSystemRolesForTenant`, `assignRoleToUser`) is used
by the dev seed. The e2e fixtures mirror it in raw SQL
(`ensureAllSystemRoles`, `grantSystemRole`); `createTenantWithAdmin` now
seeds a tenant's full system-role set and grants the admin `tenant_admin`
by default, so existing document tests (which assume full access) stay
green. `grantAdminRole: false` creates a role-less user for the
deny-path tests.

## Consequences

**Positive:**

- TD-003 retired ‚Äî within-tenant access control exists; tenant is no
  longer the only boundary. Verified live: seeded admin (tenant_admin)
  gets 200 on protected routes + full document CRUD; unauthenticated 401;
  role-less / under-privileged 403.
- The sequencing gate is satisfied ‚Äî **new domain modules (P1.5 Incidents,
  P2.x) can now ship with `@Authorize` from day one.**
- Permission strings and seeded rows share one source (`@cmc/contracts`),
  so they can't drift.
- Every denial is audited (`rbac.access.denied`, durable), feeding the
  audit-monitoring story.
- Full suite 92/92 (13 suites; +9 RBAC, documents adapted), workspace
  typecheck + lint clean.

**Negative / known gaps:**

- **No ABAC / OPA.** ToR ¬ß3.3/¬ß6.2 calls for attribute policies (Rego)
  for decisions role membership can't express ("approver may approve only
  if amount < their limit AND not own creator"). P1.1 is RBAC only; ABAC
  is a later layer (the `enforce` chokepoint is where it would attach).
- **No role/permission editing API beyond assign/remove.** Creating custom
  (non-system) roles or editing a role's permissions is not exposed yet ‚Äî
  belongs in the Admin Panel (P1.4). The tables support it.
- **`/health/deep` role-gate still pending** ‚Äî ADR-0015 deferred it to
  "once RBAC lands." Now that it has, swapping its `JwtAuthGuard` for an
  `@Authorize('audit:read')`-style gate is a one-line follow-on (not done
  in P1.1 to keep scope tight).
- **No per-permission decision-cache metrics** ‚Äî cache hit/miss isn't in
  Prometheus yet (P0.7 covers HTTP/DB; authz cache is a future counter).
- **System-role immutability is convention** (`is_system` flag) ‚Äî not yet
  enforced by a DB trigger or a guard on a (future) role-edit endpoint.
- **Resolve opens a short tenant tx per uncached request** ‚Äî one extra
  transaction on the hot path when the cache misses. Bounded by the cache
  hit rate (TTL 300s); acceptable, and the same tradeoff as the session
  cache.

## Triggers for re-evaluation

- A decision needs resource/context attributes ‚Üí add an ABAC layer (OPA
  sidecar or in-process Rego) invoked from `RbacService.enforce`.
- Admin Panel lands (P1.4) ‚Üí expose role CRUD + permission editing + a
  `delTenant` cache invalidation on role-permission changes; add the
  system-role immutability guard.
- Authz cache effectiveness in question ‚Üí add hit/miss counters to the
  P0.7 Prometheus registry.
- `/health/deep` ‚Üí gate it on a permission now that RBAC exists.
- Permission count grows large ‚Üí consider a bitmap/bloom encoding for the
  cached set (ToR ¬ß3.14 mentions this for high-cardinality permission
  sets at scale).
